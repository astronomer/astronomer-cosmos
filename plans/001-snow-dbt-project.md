## Summary

This proposal adds experimental support for running a [dbt Project on Snowflake](https://docs.snowflake.com/en/user-guide/data-engineering/dbt-projects-on-snowflake-understanding-dbt-project-objects) through Cosmos while preserving Airflow's model-level graph and observability.

The proposed integration adds a new Cosmos execution mode, `ExecutionMode.WATCHER_SNOW_DBT_PROJECT`, and uses a watcher-style architecture:

1. It renders an Airflow `DAG` or `TaskGroup` from a manifest fetched from Snowflake, with one Airflow task per dbt model or test, as in other Cosmos modes.
2. It executes the project in Snowflake through a single `EXECUTE DBT PROJECT <project_name> ARGS='build ...'` command and projects per-model statuses into the Airflow UI with near-live latency using Snowflake event telemetry. It builds on [Cosmos's watcher architecture](https://astronomer.github.io/astronomer-cosmos/guides/run_dbt/airflow-worker/watcher-execution-mode.html), introduced in Cosmos 1.11 and considered stable starting with 1.15, but requires Snowflake-specific producer and consumer classes.

Airflow remains the single control plane — cross-system dependencies, Asset-based scheduling, per-model observability, callbacks, and lineage — while Snowflake owns authoring, access control, and compute.

The integration is intentionally experimental. Snowflake documents the execution, history, live telemetry, and artifact primitives needed for this design, including how to query telemetry log entries and trace spans while a project is running. The remaining telemetry question is whether those records provide a stable programmatic vocabulary for reliable model- and test-level state.

## Motivation

Without a Cosmos integration to run dbt Projects in Snowflake, users must choose between:

- triggering a Snowflake dbt Project through a Bash operator and representing it as a single task in Airflow;
- implementing a custom operator and monitoring integration; or
- moving execution back to an Airflow worker even when the project is intended to execute in Snowflake.

Cosmos currently offers two valuable but different approaches to run dbt:

- Traditional per-node execution gives every dbt model or test its own Airflow task, with strong task-level visibility and control, but invokes dbt repeatedly, as in the default [local execution](https://astronomer.github.io/astronomer-cosmos/guides/run_dbt/airflow-worker/local-execution-mode.html).
- [Watcher execution](https://astronomer.github.io/astronomer-cosmos/guides/run_dbt/airflow-worker/watcher-execution-mode.html) runs one `dbt build` and projects its node-level progress into Airflow, reducing repeated invocation overhead while retaining a model-level Airflow graph.

Both approaches support executing dbt commands within Airflow worker nodes or containers. Cosmos can offer Snowflake-native execution while preserving Airflow as the orchestration layer for dependencies before and after dbt, and the same approach could be extended to dbt Cloud (dbt Platform).

## What dbt Projects on Snowflake provide today (based on docs)

All statements below are verified against Snowflake's public documentation as of July 2026.

1. **Execution:** `EXECUTE DBT PROJECT <name> [ARGS='<dbt command + options>']` — a blocking SQL statement. `ARGS` accepts real dbt CLI strings, including `--select` (with graph selectors), `--target`, `--vars`, and `--full-refresh`. The result row carries `SUCCESS`, `EXCEPTION`, `STDOUT`, and `OUTPUT_ARCHIVE_URL`. Jobs can be triggered asynchronously through the Python connector's `execute_async()` (`snow dbt execute --run-async` uses it).
2. **Artifacts:** [`SYSTEM$LOCATE_DBT_ARTIFACTS(query_id)`](https://docs.snowflake.com/en/user-guide/data-engineering/dbt-projects-on-snowflake-monitoring-observability) returns a `snow://dbt/…/results/…` stage path containing the full `target/` directory (`manifest.json`, `run_results.json`, and compiled SQL), retrievable via `LS`/`GET`/`COPY FILES`. Retention is 14 days, and artifacts may be absent for canceled or timed-out runs.
3. **Telemetry:** after monitoring is enabled on the schema, each execution emits OpenTelemetry log entries and trace spans to the account's active event table (`SNOWFLAKE.TELEMETRY.EVENTS` by default). [Snowflake documents querying the event table during a live dbt project execution](https://docs.snowflake.com/en/user-guide/data-engineering/dbt-projects-on-snowflake-monitoring-observability#query-the-event-table-for-dbt-project-execution-events), with entries appearing up to 10 seconds after the event occurs. These are Snowflake telemetry records, not a raw live stream of `logs/dbt.log`. They correlate with the execution through `RESOURCE_ATTRIBUTES:"snow.query.id"`; log entries use the `snow.dbt.logger` scope, and trace spans expose step names and duration. The remaining unknown is whether those messages and span names provide a stable mapping to dbt model and test lifecycle states — see [Validation items](#validation-items).
4. **Permissions:** the dbt command runs with the privileges of the `profiles.yml` target role, [intersected with](https://docs.snowflake.com/en/user-guide/data-engineering/dbt-projects-on-snowflake-access-control) the privileges of the Snowflake user calling `EXECUTE DBT PROJECT`.
5. **Concurrency:** running multiple `EXECUTE DBT PROJECT` commands concurrently against the *same project object* is [documented as unsupported](https://docs.snowflake.com/en/user-guide/data-engineering/dbt-projects-on-snowflake-limitations) today (dbt-internal `threads: N` is supported and encouraged). This constraint shapes the retry design below.
6. **Currently unsupported via `EXECUTE`:** `dbt retry`, `--state`/`--target-path`/`--log-format`/`--log-path` flags, `env_var()`, and `dbt docs serve`.
7. **Project-snapshot versioning is not yet a settled execution contract:** project objects have immutable `VERSION$n` snapshots, and [`SHOW VERSIONS IN DBT PROJECT`](https://docs.snowflake.com/en/sql-reference/sql/show-versions-in-dbt-project) identifies the snapshot targeted by the object's default pointer through `is_default`. The default can represent `LAST`, `FIRST`, or a version alias; it is not necessarily the latest snapshot. The reviewed `EXECUTE DBT PROJECT` syntax does not expose a `VERSION$n` or alias selector. Its `DBT_VERSION` parameter selects the dbt Core/Fusion runtime, not the project snapshot. Phase 0 must prove which project snapshot `EXECUTE` uses and whether it can be pinned before the skew guard is treated as sound.

## Proposed design

### 1. Rendering — manifest fetched from Snowflake

**Principle: the user's CI creates the manifest; Cosmos fetches it.** The manifest is compiled by the project's CI and deployed alongside the Snowflake dbt project. At render time, Cosmos resolves the project's default `VERSION$n` snapshot and fetches the manifest from that exact snapshot. This avoids generating a manifest during DAG parsing, but it does not by itself prove that a later `EXECUTE` will use the same snapshot.

- **Path: the project's manifest:** users deploy `manifest.json` alongside the dbt project files. The deployment makes it part of a DBT PROJECT snapshot, addressable through the `location_uri` returned by `SHOW VERSIONS IN DBT PROJECT`, for example `snow://dbt/<DB>.<SCHEMA>.<OBJECT>/versions/version$n/…/manifest.json`. Version-file read behavior and retention still require validation; Cosmos must not substitute the 14-day execution-artifact location for this render input.
- **Mechanism — extend `ObjectStoragePath` with a Snowflake backend:** Cosmos currently loads remote manifests from `s3://`, `gs://`, and `abfs://` through Airflow's `ObjectStoragePath`. Snowflake's [`SnowFileSystem`](https://github.com/snowflakedb/snowflake-ml-python/blob/f0c5cdd18481f39ab6b7b657b5ea84c573e7974f/snowflake/ml/fileset/snowfs.py) already provides the required read-only fsspec implementation for `snow://` embedded-stage paths. Airflow does not preconfigure that backend today, but its [alternative-backend API](https://airflow.apache.org/docs/apache-airflow/2.11.0/core-concepts/objectstorage.html#alternative-backends) supports attaching an fsspec filesystem to a protocol. Phase 1 therefore adds a Cosmos compatibility helper that resolves `manifest_conn_id` through `SnowflakeHook`, constructs `SnowFileSystem`, and calls `attach(protocol="snow", conn_id=manifest_conn_id, fs=filesystem)` **before** constructing or deserializing the `ObjectStoragePath`. The existing manifest loader can then read the versioned `snow://` path and populate its cache. A direct connector downloader remains a fallback if the path integration fails validation; upstreaming the attachment into the Airflow Snowflake provider would make it available outside Cosmos.
- **Caching is mandatory:** downloading on every DAG parse is a non-starter at the scale users have described (hundreds of projects per Airflow deployment). The cache key combines the default snapshot identity (`name`/`location_uri` plus `git_commit_hash` when present from `SHOW VERSIONS IN DBT PROJECT`) with the compile fingerprint (`target_name` + render-affecting `--vars`). A manifest is compiled *for* a target, so snapshot identity alone is not a sufficient key. Refreshes are staggered or deploy-triggered (the user's CI requests a refresh as its final deployment step), never performed per parse.
- **Skew guard — gated on Phase 0:** immediately before submission, the producer resolves the current default snapshot and compares it with the snapshot in the rendered cache key. A mismatch fails before execution or triggers an explicit re-render workflow; it never silently proceeds. Phase 0 must prove that this default pointer is also the snapshot selected by `EXECUTE`. If that correspondence cannot be established, the model-level mode does not ship.
- **Load-mode support is explicit:** this mode supports **manifest-based loading only**, after the mode-specific fetch materializes the manifest locally. `LoadMode.DBT_LS` (and `DBT_LS_FILE`/`CUSTOM`) are intentionally out of scope — a remote `dbt ls` on every refresh would add an `EXECUTE` round-trip under the same concurrency constraints while yielding strictly less information than the manifest. Cosmos raises `CosmosValueError` at render time.

### 2. Execution — the watcher pattern over Snowflake

*(Naming: `WATCHER_SNOW_DBT_PROJECT` so that `snow` matches Snowflake's own CLI.)*

- **Submission and idempotency:** before the first submission, the producer derives a deterministic execution identity from the dbt project FQN plus the Airflow `dag_id`, `run_id`, `task_id`, and `map_index`, and sets it as Snowflake's `QUERY_TAG`. Before *every* call to `execute_async()`, including attempt 1, it searches the low-latency Information Schema [`QUERY_HISTORY`](https://docs.snowflake.com/en/sql-reference/functions/query_history) for that tag and performs a separate project-level in-flight check. If the tagged query already exists, the producer adopts or reconciles it. If it observes a different active query targeting the same project object — from another DAG, backfill, Task, or Snowsight — the producer fails without submitting. Only the no-match/no-conflict path may submit a new `EXECUTE DBT PROJECT`; Phase 0 must characterize the residual check-and-submit race.
- **Producer:** after the pre-submit checks, one task submits `EXECUTE DBT PROJECT <obj> ARGS='build …'` through the connector's `execute_async()` and captures the query ID for polling, artifacts, and cancellation. It polls the event table approximately every five seconds and progressively pushes model and test start/completion telemetry onto watcher-compatible XCom keys. Because Snowflake documents an event-ingestion delay of up to 10 seconds, expected status latency is approximately 10–15 seconds rather than one polling interval. After the query becomes terminal, the same task harvests and reconciles the execution artifacts before publishing the artifact-ready barrier used by model consumers. Like the local watcher's producer, the task is synchronous and stays on its worker for the run's duration; a deferrable variant is a possible later optimization, at the cost of the near-live updates that justify the design.
- **Consumers:** Snowflake-specific consumer subclasses reuse `BaseConsumerSensor`, the watcher XCom-key helpers, and aggregation utilities, but do **not** inherit the local `DbtConsumerWatcherSensor` unchanged. Current watcher consumers require `project_dir` and inherit a transport-specific dbt operator; `WATCHER_KUBERNETES` likewise has its own consumer hierarchy. The Snowflake consumers must remove the local-project requirement, implement Snowflake fallback behavior, and fail closed on unexplained missing node statuses.
- **Post-run reconciliation is authoritative and fail-closed:** on completion, the producer fetches `run_results.json` and reconciles every node expected from the rendered manifest and selection. Explicitly classified ephemerals or known empty nodes may retain no-op semantics; any other rendered node missing from the executed results is an error. The producer fails and the Snowflake consumers fail rather than taking the local watcher's `NODE_NOT_RUN` success branch. This prevents render/execute skew or selector drift from producing a green DAG.
- **XCom shape, not implementation, is reused:** the poller keeps the watcher-compatible `<uid>_status` dictionaries, aggregated `<model_uid>_tests_status`, `task_status`, and startup-event keys so shared sensor utilities remain useful. The Snowflake mode changes missing-status semantics and uses different producer/consumer classes. Upstream-failure skips are rewritten to `failed`, never `skipped` (consistent with [#2698](https://github.com/astronomer/astronomer-cosmos/issues/2698)).
- **Compiled SQL remains an output-only templated field on each model task, but it is post-run for this transport:** to preserve [current Cosmos behavior](https://astronomer.github.io/astronomer-cosmos/guides/cosmos_devex/compiled-sql.html), Cosmos does not assume that the render-time manifest contains compiled SQL or that Snowflake exposes execution artifacts while `EXECUTE` is still running. After the remote command becomes terminal, the producer calls [`SYSTEM$LOCATE_DBT_ARTIFACTS(query_id)`](https://docs.snowflake.com/en/sql-reference/functions/system_locate_dbt_artifacts) and reads the raw execution `manifest.json`. For each selected model unique ID, it reads `manifest["nodes"][uid]["compiled_code"]` and pushes that value through the existing watcher helper to the canonical `<uid>_compiled_sql` XCom key. Reading the raw artifact at runtime avoids adding all compiled SQL to `DbtNode` or the serialized DAG. The producer then publishes an explicit artifact-harvest-complete barrier. A Snowflake model consumer that has already observed a terminal status waits for that barrier, pulls its own SQL, updates its rendered-task-instance fields, and only then returns success or failure, so both successful and failed model tasks can show SQL under **Rendered Templates**. The barrier also distinguishes “not harvested yet” from “harvest complete but SQL unavailable.” The producer does not publish an aggregate compiled-SQL field. `compiled_sql` cannot be supplied through `operator_args`, and `should_store_compiled_sql=False` skips the wait and publication. If artifacts are unavailable, a selected unique ID is missing, or `compiled_code` is null, the producer publishes the completed barrier with a reason; the consumer leaves the field empty, emits an actionable warning, and preserves the authoritative dbt result.
- **Cancellation:** `on_kill` → `SYSTEM$CANCEL_QUERY(query_id)`; canceled runs may leave no artifacts (documented Snowflake behavior).
- **Resilience — query-tag discovery, then reattach or reconcile:** if the producer dies ungracefully, the `EXECUTE` keeps running server-side. [Airflow clears task XComs on retry](https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/xcoms.html), and a worker can die after `execute_async()` but before pushing the query ID, so XCom is only a convenience checkpoint. The deterministic `QUERY_TAG` in Snowflake is the durable recovery identity: on retry, the producer discovers the query from history, adopts it if active, or reconciles it if terminal. `ABORT_DETACHED_QUERY = false` is set explicitly so worker death leaves the query adoptable.
- **Producer ARGS mapping:** `RenderConfig.select`/`selector`/`exclude` (plus `vars`, `full_refresh`, `target_name`) serialize into the single `build` command exactly as the local watcher forwards them. Invariant: the rendered graph and the executed set derive from the same manifest version and byte-identical selection expressions.

### 3. Retries

Retries are watcher-compatible at the Airflow graph level but transport-specific in implementation:

- A **producer retry never blindly submits** and does not use the local watcher's restore-and-skip branch. It re-runs query-tag discovery: an active tagged query is adopted, and a terminal tagged query is reconciled from telemetry and artifacts. If no tagged query exists, submission is allowed only after the project-level in-flight and version checks pass. Reconciliation rebuilds the consumer-visible XComs that Airflow cleared.
- A **consumer retry requires a Snowflake-specific override** of `_fallback_to_non_watcher_run`; it cannot inherit the local or Kubernetes fallback. The intended command is a per-node `EXECUTE … ARGS='run --select <node>'` using the Snowflake connection identified by `manifest_conn_id` and the same pre-submit conflict check.
- **The consumer fallback is disabled in v1:** parallel per-model fallbacks would issue concurrent `EXECUTE` commands against one project object, which is unsupported today. Clearing or retrying a consumer therefore fails with an actionable message directing the user to start a new DAG run. The fallback can be enabled only if Snowflake supports the required concurrency and the transport-specific implementation passes Phase 4 validation.

### 4. Target config surface

The following is the intended post-implementation API. It does **not** parse on current `main`: `validate_changed_config_paths` currently requires `ExecutionConfig.dbt_project_path` in every mode, and the watcher task builder currently passes that path as `project_dir` to all consumers. The contract changes required to make this example valid are registered in [Compatibility and required contract changes](#compatibility-and-required-contract-changes).

```python
DbtDag(
    project_config=ProjectConfig(
        manifest_path="snow://dbt/ANALYTICS_DB.DBT.PROJECT_A/versions/version$42/target/manifest.json",
        manifest_conn_id="project_a_conn",  # Snowflake credentials for rendering and execution
        project_name="project_a",  # required with manifest_path, as for other remotes
    ),
    render_config=RenderConfig(
        select=["+revenue"],  # optional → how Cosmos exposes '--select +revenue'
    ),
    execution_config=ExecutionConfig(
        execution_mode=ExecutionMode.WATCHER_SNOW_DBT_PROJECT,
    ),
    # Optional: ARGS='--target prod'; if omitted, the object's DEFAULT_TARGET applies.
    operator_args={
        "target_name": "prod",
    },
)
```

- **The canonical manifest URL identifies the execution object.** Cosmos parses and validates the DBT PROJECT FQN (`ANALYTICS_DB.DBT.PROJECT_A` above) from `snow://dbt/<FQN>/versions/<VERSION$n>/…` and passes that object to `EXECUTE`. This keeps render and execution identity tied to one versioned source. Noncanonical manifest locations and an explicit object override are alternatives rather than the starting contract.
- **`manifest_conn_id` is the mode's Snowflake connection.** Current `ProjectConfig` uses the argument only to construct `ObjectStoragePath` and does not retain or forward it to operators. This mode changes that contract: `ProjectConfig` persists the resolved connection ID, and the converter passes it to the Snowflake producer and consumers for both manifest reads and execution. Separate render and execution connections remain a possible later extension.
- **`ProfileConfig` is optional in this mode** (precedent: the container-based modes). The canonical `profiles.yml` lives *inside* the Snowflake project — it defines targets and carries no credentials (identity comes from the executing session). Cosmos fetches it alongside the manifest (same version-keyed cache) to validate `target_name` and derive dataset/OpenLineage namespaces.
- **Monitoring is a prerequisite:** Snowflake requires `LOG_LEVEL`, `TRACE_LEVEL`, and `METRIC_LEVEL` to be configured on the schema that contains the dbt project object before telemetry log entries and trace spans are emitted. Cosmos should validate the active monitoring configuration at task start and fail with actionable guidance; it should not alter schema-level settings.
- **Render-time guardrails:** Cosmos raises `CosmosValueError` when a configuration unsupported by this mode is passed — a non-manifest `LoadMode`, `profile_mapping`, `TestBehavior` other than `AFTER_EACH` (pending validation of `--resource-type` through `EXECUTE`), or `source_rendering_behavior != NONE` (`source freshness` is not in Snowflake's supported-command list). `max_active_runs=1` is recommended as defense in depth, but it protects only one DAG object; the runtime project-level in-flight check is mandatory.

### Credentials

The mode authenticates through the Airflow Snowflake connection named by `ProjectConfig.manifest_conn_id` for both manifest access and `EXECUTE DBT PROJECT`. Per Snowflake's documented access model, the calling role's privileges intersect with the `profiles.yml` target role. That connection therefore needs the read/monitor privileges required for the manifest plus USAGE on the project and warehouse and the relevant execution privileges. Scoping connections by team or project is an Airflow deployment concern, not a Cosmos concern. A keyless option (Workload Identity Federation) becomes available to Cosmos when [dbt-snowflake ships WIF support](https://github.com/dbt-labs/dbt-adapters/milestone/6); the Airflow Snowflake provider has supported it since [6.14.0](https://github.com/apache/airflow/pull/68107). A `SnowflakeWorkloadIdentityProfileMapping` is planned as separate follow-up work.

## Alternatives considered

1. **Per-model `EXECUTE … --select <node>` fan-out** ("Cosmos classic, compute in Snowflake"): concurrent `EXECUTE` commands against one project object are unsupported today, and serialized execution multiplies wall-clock time and warehouse-resident time because each model run pays the full `EXECUTE` startup overhead. This approach is ruled out as the primary path and is only a future candidate for consumer retries (§3); it remains disabled in v1. It could be revisited if Snowflake's concurrency semantics change.
2. **Thin whole-run trigger operator** (trigger, poll, and fetch artifacts, like existing SaaS dbt operators): this could ship in weeks, and most of its plumbing (artifact client and history poller) is prerequisite work for this proposal. However, as a product answer, it produces a single opaque task, losing the per-model observability, retries, and lineage that are Cosmos's core value. It is kept as internal plumbing, not the headline.
3. **Trigger-only via Snowflake Tasks** (an OPERATE-only role fires pre-created Tasks that wrap `EXECUTE`; Tasks run with the owner's privileges, so the trigger role holds zero data access): this is now a viable alternative to prototype. Snowflake documents [`EXECUTE TASK … USING CONFIG`](https://docs.snowflake.com/en/sql-reference/sql/execute-task) for overriding JSON configuration for a single execution without changing the Task definition; the Task can read those values through [`SYSTEM$GET_TASK_GRAPH_CONFIG`](https://docs.snowflake.com/en/sql-reference/functions/system_get_task_graph_config). Airflow could therefore pass selectors, target, and vars dynamically instead of freezing them in the Task definition. However, Snowflake serializes triggers while a Task is running, and repeated triggers replace the pending configuration with the most recent one. A single shared Task is therefore unsafe for overlapping Airflow runs with different parameters unless Cosmos serializes them or provisions separate Tasks. Other tradeoffs are a pre-created Task per project, owner-role security semantics, and additional work to correlate, cancel, and reattach to the underlying `EXECUTE DBT PROJECT` query.
4. **Separate render and execution configuration:** add `snowflake_conn_id` and `snowflake_project` as operator arguments or typed fields. This supports least-privilege deployments with different identities for artifact access and execution, and manifests stored outside the canonical project snapshot. It is not the starting API because it duplicates values already carried by `manifest_conn_id` and the canonical `snow://dbt/<FQN>/…` URL. It can be added later as an explicit override without changing the default single-connection contract.

## Known v1 limitations

These combine Snowflake platform constraints with deliberate limits of the initial Cosmos integration:

- **One `EXECUTE` at a time per dbt project object** — so per-model consumer retries are gated (§3). A fresh DAG run does not bypass an active execution: every submission performs the same project-level in-flight check.
- **Live telemetry is delayed:** Snowflake documents up to 10 seconds of event-ingestion delay. The messages and trace spans are available during execution, but their stability as a model/test lifecycle API remains unvalidated.
- **`on_warning_callback` support is initially degraded:** the event table supports filtering `WARN` and `ERROR` log entries, but whether dbt test warnings have a stable, parseable message shape is unvalidated. Warnings may still need post-run reconstruction from `run_results.json`.
- **dbt docs:** `dbt docs serve` is unavailable in Snowflake, but `docs generate` works via `EXECUTE` — artifacts can be pulled from the stage and then handed to Cosmos's dbt-docs serving path through new artifact plumbing.
- **Artifacts are retained for 14 days** and may be absent for canceled or timed-out runs — Cosmos harvests them immediately after each run.
- **Compiled-SQL parity delays model-task completion:** with `should_store_compiled_sql=True`, live telemetry can still report per-model progress, but model consumers wait for the overall remote command and artifact harvest before becoming terminal in Airflow. Setting the flag to `False` permits near-live terminal model states at the cost of an empty rendered `compiled_sql` field. Earlier access to compiled artifacts would be an optimization only if Phase 0 proves that Snowflake exposes them safely during execution.
- **Project-snapshot selection is unresolved:** the reviewed `EXECUTE` syntax has no explicit `VERSION$n` selector, but that alone does not prove which snapshot Snowflake executes or that pinning is impossible. The mode cannot ship until Phase 0 establishes the behavior and a sound skew policy.
- **`env_var()` is unsupported** in Snowflake-executed projects — use `--vars`.

## Phased delivery

| Phase | Scope | Exit criteria |
|---|---|---|
| **0. Validation** | Run the spike checklist below against a live Snowflake account | project-snapshot selection, telemetry vocabulary, recovery discovery, and concurrency behavior have explicit go/no-go answers for Phases 1–2 |
| **1. Render** | `snow://` backend attachment for `ObjectStoragePath`, exact-snapshot manifest cache, config-contract changes, and render-time guardrails | the §4 example parses without a local project across the supported Airflow matrix; a representative project renders in the Airflow UI; parse-time cost is measured and acceptable; unsupported configs fail fast |
| **2. Execute** | Snowflake-specific producer and consumers, async `EXECUTE`, telemetry poller, query-tag recovery, mandatory project-level in-flight check, fail-closed reconciliation, and cancellation | per-model telemetry appears near-live; a missing expected node cannot resolve green; recovery after worker death between submission and XCom push does not duplicate execution; two DAGs and an external Snowsight/Task run cannot pass the guard silently |
| **3. Artifacts & parity** | terminal artifact harvest, compiled-SQL readiness barrier, per-model `compiled_sql` output-only template fields, `run_results` XCom, callback re-feed shim, dbt docs pull, OpenLineage/Asset emission | successful and failed model tasks wait for artifact reconciliation and show their own compiled SQL in Airflow's Rendered Templates view across Airflow 2.x/3.x; unavailable artifacts release consumers with an actionable warning rather than hanging; callbacks + lineage work; remaining gaps vs local mode are documented |
| **4. Retry & concurrency** *(gated on Snowflake lifting the same-object concurrency limitation and supporting `dbt retry` via `EXECUTE`)* | per-model consumer retries, native `dbt retry` | retrying a failed model from the Airflow UI re-runs only that model |

Phases 1–3 would ship behind an experimental flag, following the precedent of previous execution-mode introductions.

## Validation items

Empirical questions to settle against a live Snowflake account before or during Phases 1 and 2. Items 1–4 are ship gates for model-level execution:

1. **Project-snapshot selection and pinning:** execute while the default pointer changes between two `VERSION$n` snapshots. Establish which snapshot `EXECUTE` selects, when it resolves that selection, whether any documented pinning mechanism exists, and whether `SHOW VERSIONS … is_default` is a valid pre-submit skew guard. If render and execution cannot be bound to the same snapshot, do not ship the model-level mode.
2. **Per-node event vocabulary and fail-closed reconciliation:** Snowflake documents live log and trace queries, correlation through `RESOURCE_ATTRIBUTES:"snow.query.id"`, and up to 10 seconds of ingestion delay. Validate which records reliably identify model/test start and finish, dbt unique ID, terminal status, and test warnings across supported commands and adapters. Deliberately render a node not present in `run_results.json` and verify that the producer and consumer fail rather than taking `NODE_NOT_RUN` as success.
3. **Crash-window recovery:** kill the producer after `execute_async()` returns but before any XCom push. Verify that the deterministic `QUERY_TAG` is immediately discoverable through Information Schema `QUERY_HISTORY`, that retry adopts exactly one query, and that terminal-query reconciliation recreates cleared XCom state. If query history is not sufficiently prompt, identify another durable server-side discovery mechanism before Phase 2.
4. **Cross-orchestrator concurrency:** race two independent DAGs, a backfill, and a Snowsight or Snowflake Task invocation against the same project. Validate the project-level in-flight lookup and Snowflake's response to simultaneous submissions. Document the check-and-submit time-of-check/time-of-use window; if it cannot be made safe enough or fail closed, require dedicated project objects or another serialization mechanism rather than relying on `max_active_runs`.
5. **`ObjectStoragePath` backend integration and performance:** attach `SnowFileSystem` under `snow` using the Airflow 2.x and 3.x import paths, then validate construction, serialization/deserialization, process-local registration, connection lookup, and reads across the supported Airflow matrix. Compare it with direct connector `GET` and `COPY FILES` → stage → `GET`; measure manifest fetch latency against the DAG-parse budget and determine whether a MONITOR-only role is sufficient for reads.
6. **Version-file downloads:** validate reading `snow://dbt/<obj>/versions/…` through the attached `SnowFileSystem` backend and connector alternatives. This gates the primary manifest path; the git repository stage is a documented fallback route to the same CI-produced file.
7. **Compiled SQL rendered-template parity and artifact timing:** establish when `SYSTEM$LOCATE_DBT_ARTIFACTS` and the execution's `manifest.json` become readable; v1 must work when the manifest is available only after terminal completion. Validate `nodes[uid].compiled_code`, selected unique IDs that are missing or have null compiled code, producer ordering (`<uid>_compiled_sql` before the artifact-ready barrier), consumer waiting after an early terminal telemetry event, successful and failed consumers, unavailable/canceled artifacts, deferrable and non-deferrable paths, Airflow 2.x and 3.x rendered-task-instance persistence, retry reconstruction, and `should_store_compiled_sql=False` bypassing the barrier.
8. **Full raw-log availability after completion:** distinguish `logs/dbt.log` or `SYSTEM$GET_DBT_LOG` from event-table telemetry, and validate availability for deployed and scheduled runs.
9. **`STDOUT` result column contents** on successful, warning, and failed runs (feeds warning-callback parsing).
10. **Cancel semantics:** partial materialization and artifact presence after `SYSTEM$CANCEL_QUERY` mid-run.
11. **`--resource-type` via `EXECUTE ARGS`** — gates `TestBehavior.NONE`/`AFTER_ALL` support.
12. **`source freshness` via `EXECUTE`** — absent from the supported-commands list; gates source rendering.
13. **Task dynamic-configuration alternative:** prototype an OPERATE-only role calling `EXECUTE TASK … USING CONFIG` with selectors, target, and vars. Confirm configuration validation, owner-role execution, underlying dbt query-ID correlation, cancellation, and reattachment behavior. Verify that overlapping Airflow runs cannot silently replace one another's pending configuration; otherwise, reject this alternative or require explicit serialization.

## What we'd welcome from Snowflake

Capabilities that would close the gap between this mode and local Cosmos parity, based on the documented behavior above:

1. A documented telemetry **event schema** for dbt runs, with a guarantee that events carry the parent `EXECUTE` query ID.
2. A documented project-snapshot selector or pinning contract on `EXECUTE` — eliminates render/execute skew instead of relying on Cosmos's pre-submit guard.
3. **Concurrent execution** against the same project object through `EXECUTE` (unlocks per-model retries in Phase 4).
4. `--resource-type` and `dbt source freshness` via `EXECUTE ARGS`.
5. `env_var()` support at execution.
6. Artifact guarantees: retention beyond 14 days (or configurable retention) and artifact upload for canceled or timed-out runs.

## Compatibility and required contract changes

This is a new execution mode, but it is not implementable as a registration-only addition. The table below records every known current contract that the implementation must change. Existing modes remain behaviorally compatible only if each change is scoped to `WATCHER_SNOW_DBT_PROJECT` and covered by regression tests.

| Surface | Current contract on `main` | Required change for this mode | Compatibility guard |
|---|---|---|---|
| Execution project path | `validate_changed_config_paths` requires `ExecutionConfig.dbt_project_path` for every mode. | Exempt only `WATCHER_SNOW_DBT_PROJECT`; require an exact-snapshot canonical `snow://dbt/<FQN>/…` manifest and derive the execution object from it. | Tests prove every existing execution mode retains its current path validation; invalid or ambiguous Snowflake URLs fail at parse time. |
| Task construction | The converter includes `project_dir=execution_config.project_path` in common task arguments. | Build Snowflake-mode task arguments separately and do not pass a local project path to its producer or consumers. | Existing-mode task-argument snapshots remain unchanged. |
| Consumer classes | Local watcher consumers require `project_dir`, inherit local dbt operators, and use local fallback; `WATCHER_KUBERNETES` already uses transport-specific subclasses. | Add Snowflake-specific producer, model consumer, and test consumer classes that reuse only transport-neutral watcher utilities. | No changes to the local and Kubernetes class hierarchies or fallback behavior. |
| Missing-node semantics | `WatcherTrigger` can emit `NODE_NOT_RUN`, and current consumers treat that branch as success. | Snowflake reconciliation must fail when an expected rendered node is absent, except for explicit no-op classifications. | Keep existing semantics for existing watcher modes; add skew tests for the Snowflake mode. |
| Manifest transport | `ProjectConfig.manifest_path` uses Airflow `ObjectStoragePath`; no Snowflake filesystem is attached for the `snow` protocol today. | Before constructing the path, create `SnowFileSystem` from `manifest_conn_id` and attach it as the `snow` backend through Airflow's supported object-store API. Continue through the existing manifest loader and cache. | Scope registration to the protocol/connection pair, cover Airflow 2.x and 3.x import paths, and leave existing remote schemes unchanged. |
| Connections and object identity | `manifest_conn_id` is consumed while creating the manifest path but is not persisted on `ProjectConfig` or forwarded for execution. `ProfileConfig` is optional outside LOCAL/VIRTUALENV, and `profile_mapping` is not an execution connection for this design. | Persist the resolved `manifest_conn_id`, pass it to the Snowflake producer and consumers, and derive the project FQN from the validated canonical manifest URL. | Scope the new reuse behavior to this mode; existing modes keep their current connection behavior. Tests prove the rendered and executed project identities match. |
| Initial profile validation | `validate_initial_user_config` requires `ProfileConfig` only for LOCAL and VIRTUALENV. | No change is needed. Add separate mode-specific validation for unsupported `profile_mapping` and load modes. | Regression test the existing LOCAL/VIRTUALENV error and container-mode allowance. |
| Retry and recovery | The local producer's retry branch restores watcher state and skips; XComs are cleared by Airflow before retry. | Replace that branch in the Snowflake producer with query-tag discovery, adoption, terminal reconciliation, and XCom reconstruction. | Existing watcher retry behavior remains untouched; crash-window tests cover the new mode. |
| XCom protocol | Current consumers depend on watcher status keys and accept transport-specific producer behavior. | Reuse the key/value shape only; do not claim implementation reuse. Snowflake missing-state and recovery semantics are stricter. | Shared helpers get additive tests; no key or value changes for existing modes. |
| Rendered compiled SQL | Current local watcher can read `target/compiled` while dbt is running and expose output-only `compiled_sql` through each model consumer's `template_fields`. Snowflake execution artifacts cannot be assumed available to the Airflow worker before the remote command finishes, and the current `DbtNode` representation does not carry manifest `compiled_code`. | After terminal execution, read `nodes[uid].compiled_code` directly from the raw remote execution manifest and push each selected model's SQL under the canonical per-model XCom key without serializing the SQL into the DAG. Publish a new artifact-ready barrier only after those pushes. Snowflake model consumers with compiled-SQL capture enabled wait for that barrier before updating their own rendered fields and returning; retries repeat the manifest harvest and XCom publication. | Test ordering and no-hang behavior for successful, failed, canceled, missing-artifact, missing-node, and null-`compiled_code` runs on Airflow 2.x/3.x; preserve the near-live path when `should_store_compiled_sql=False`; never expose aggregate producer SQL or accept user-supplied `compiled_sql`. |
| Optional dependencies | Airflow does not install or attach `SnowFileSystem`, and constructing it requires a Snowflake connector connection or Snowpark session. | Add a lazy Snowflake-mode extra containing the client packages required to construct and attach `SnowFileSystem`; retain a connector-only backend as an alternative if `snowflake-ml-python` is too heavy. | Keep imports and attachment mode-scoped so users of other modes do not install, import, or initialize Snowflake-only packages. |

## Open questions (for community input)

1. Manifest/profile cache backend: scheduler-local disk or object storage — does anyone have strong operational preferences at 100+ projects per deployment?
2. Is `WATCHER_SNOW_DBT_PROJECT` the right name, or should the watcher transport be an orthogonal axis (e.g. `ExecutionMode.SNOW_DBT_PROJECT` implying watcher semantics)?
3. Should the standalone artifact-fetching plumbing (locate and download `target/` for a given query ID) also ship as public, reusable operators or hooks independent of the execution mode?
4. What customer requirements would justify adding separate render/execution connection IDs or an explicit project-object override after the single-connection API ships?
